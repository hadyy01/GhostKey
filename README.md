## Background

The Internet of Things connects billions of devices that collectively suffer from a
fundamental security problem: **they have no hardware-based identity**. A device's
"identity" is typically a password burned into flash — something an attacker with
code execution can simply read out.

During a summer internship at LDCE in 2022, I built a proof-of-concept secure
communication system for IoT devices using WebSockets and AES-CBC encryption. The
system worked, but had three structural weaknesses I identified at the time and
documented in my report:

1. **No hardware-rooted device identity** — keys came from a software PRNG.
2. **Entropy from software randomness** — `secrets.choice()` is good but physically
   observable by a privileged attacker.
3. **AES-CBC with a static IV** — a known-plaintext-leaking mode with no authentication.

This repository is the revamped version, implementing hardware-rooted solutions
to all three problems using only the silicon already present in the ESP8266.

## Three Core Technical Contributions

### Contribution 1 — SRAM Physical Unclonable Function (PUF) for Device Identity

**The problem:** How do you give a cheap microcontroller a unique, unforgeable identity
without adding external secure hardware (TPM, ATECC608, etc.)?

**The insight:** When an ESP8266 powers on, its SRAM cells settle into a pattern
determined by manufacturing variation in the NMOS pull-down transistors. This pattern
is:
- **Unique** — no two chips settle into the same pattern (inter-chip Hamming distance ≈ 50%)
- **Stable** — the same chip settles into the same pattern across power cycles (intra-chip
  Hamming distance ≈ 0% after debiasing)
- **Unclonable** — determined by sub-micron geometry that cannot be read non-destructively

The SRAM startup state is read from base address `0x3FFE8000` before the WiFi stack
initialises heap memory. Von Neumann debiasing removes systematic bit bias, and SHA-256
produces a stable 256-bit device fingerprint.

```
Power-on SRAM → Von Neumann Debias → SHA-256 → 256-bit Device Fingerprint
```

The Raspberry Pi stores fingerprints in an enrollment registry. From then on, any device
can authenticate by presenting its PUF response — without ever transmitting a secret.

**Security property:** An attacker who steals the firmware binary cannot clone the device,
because the secret is the physical silicon geometry, not software.

**Reference:** Maiti et al., *"A Systematic Method to Evaluate and Compare the Performance of Physical Unclonable Functions"*, IACR ePrint 2011/657.

### Contribution 2 — Ring-Oscillator True Random Number Generator (TRNG)

**The problem:** Session keys generated by `secrets.choice()` on the Raspberry Pi are
cryptographically strong in software, but the entropy source is the OS CSPRNG — an
attacker with root access can potentially predict it. More importantly, the device itself
should be generating its own key material using physical entropy that is unobservable
from software.

**The insight:** The ESP8266 contains two independent oscillators — the CPU PLL (80 MHz)
and the WiFi modem PLL (~80 MHz). Reading the Xtensa CPU cycle counter (`ccount` register)
and XOR-sampling it against `micros()` (which uses a different hardware timer) captures
thermal jitter between the two oscillators. This jitter is:

- **Physically based** — determined by thermal noise in the oscillator resistors
- **Fast** — 128 samples collected in ~130 µs
- **Unobservable** — an attacker with code execution cannot predict or replay thermal noise

Samples are von Neumann debiased and hashed through SHA-256, producing a uniform 128-bit
session key each boot.

```
CPU ccount XOR micros() → 128 samples → Von Neumann Debias → SHA-256 → 128-bit session key
```

The device encrypts this session key and sends it to the Raspberry Pi, which uses it
as the decryption key for all subsequent messages in that session.

**Reference:** Sunar et al., *"A Provably Secure True Random Number Generator with Built-In Tolerance to Active Attacks"*, IEEE Transactions on Computers, 2007.

---

### Contribution 3 — Upgrading AES-CBC to AES-128-GCM

**The problem:** The original system used AES-128-CBC with a hardcoded IV
(`'ABCDABCDABCDABCD'`). This has two critical weaknesses:

| Property | AES-CBC (original) | AES-128-GCM (new) |
|---|---|---|
| Same message → same ciphertext? | **Yes** (static IV leaks this) | No (fresh nonce per message) |
| Detects ciphertext tampering? | **No** | Yes (GCM tag) |
| Security model | Not IND-CPA secure | IND-CCA2 secure |
| Extra overhead on ESP8266 | Baseline | +16 bytes (tag) +12 bytes (nonce) |

GCM (Galois/Counter Mode) combines a stream cipher (CTR mode) with a polynomial MAC
(GHASH). Any single bit-flip in the ciphertext causes the 128-bit tag verification
to fail, and decryption is rejected **before** any plaintext is produced.

The sensor data JSON is also passed as Additional Authenticated Data (AAD), meaning
the device metadata is authenticated but not encrypted — useful for routing and
logging without decryption overhead.

**Wire format:**
```
[ 12-byte GCM nonce | variable-length ciphertext | 16-byte GHASH tag ]
→ base64-encode → transmit as WebSocket text frame
```

**Reference:** Dworkin, *NIST Special Publication 800-38D: Recommendation for Block Cipher Modes of Operation: Galois/Counter Mode (GCM)*, 2007.

## System Architecture

```
┌──────────────────────────────────────────────────────────────────────┐
│                        Local WiFi Network / WebSocket TCP            │
│                                                                      │
│  ┌────────────────────────┐              ┌────────────────────────┐  │
│  │   ESP8266 / NodeMCU    │              │    Raspberry Pi 3/4    │  │
│  │      (IoT Client)      │              │   (Key + Data Server)  │  │
│  │                        │              │                        │  │
│  │  ┌──────────────────┐  │              │  ┌──────────────────┐  │  │
│  │  │  [C1] SRAM PUF   │  │──AUTH:fp──▶ │  │  Key Server      │  │  │
│  │  │  Device identity │  │ ◀──KEY:k─── │  │  :9000           │  │  │
│  │  └──────────────────┘  │              │  │  PUF registry    │  │  │
│  │                        │              │  │  session_store   │  │  │
│  │  ┌──────────────────┐  │              │  └──────────────────┘  │  │
│  │  │  [C2] RO-TRNG    │  │              │                        │  │
│  │  │  Session entropy │  │              │  ┌──────────────────┐  │  │
│  │  └──────────────────┘  │              │  │  Data Server     │  │  │
│  │                        │──GCM msg──▶  │  │  :8010           │  │  │
│  │  ┌──────────────────┐  │ ◀───ACK───── │  │  GCM decrypt     │  │  │
│  │  │  [C3] AES-128-GCM│  │              │  │  sensor log      │  │  │
│  │  │  Authenticated   │  │              │  └──────────────────┘  │  │
│  │  │  encryption      │  │              │                        │  │
│  │  └──────────────────┘  │              └────────────────────────┘  │
│  └────────────────────────┘                                          │
└──────────────────────────────────────────────────────────────────────┘
```

### Protocol flow (per boot, step by step)

| Step | Actor | Action |
|------|-------|--------|
| 1 | ESP8266 | Reads raw SRAM from `0x3FFE8000` before heap init. Applies von Neumann debiasing. Hashes with SHA-256 → 256-bit device fingerprint. |
| 2 | ESP8266 → Key Server | Sends `AUTH:<64-hex-fingerprint>` over WebSocket to port 9000. |
| 3 | Key Server | Looks up fingerprint in `device_registry.json`. If matched: generates fresh 128-bit key via `os.urandom(16)`. Writes to `session_store.json`. |
| 4 | Key Server → ESP8266 | Sends `KEY:<32-hex-key>`. Key channel closed. |
| 5 | ESP8266 | For each sensor reading: samples `ccount XOR micros()` × 128 → SHA-256 → 12-byte fresh GCM nonce. Encrypts with AES-128-GCM. |
| 6 | ESP8266 → Data Server | Sends base64 payload to port 8010. |
| 7 | Data Server | Fetches session key from `session_store.json`. Verifies GCM tag → decrypts → logs sensor JSON. |

### Wire format (each encrypted WebSocket frame)

```
┌─────────────────┬──────────────────────────────┬──────────────────┐
│   12 bytes      │         variable              │    16 bytes      │
│   GCM nonce     │   Ciphertext (sensor JSON)    │   GHASH tag      │
│  (fresh/msg)    │                               │  (tamper detect) │
└─────────────────┴──────────────────────────────┴──────────────────┘
                         │
                         ▼  base64-encode
                  WebSocket text frame
```

### Security comparison: original (2022) vs this system

| Threat model | Original AES-CBC (2022) | This system |
|---|---|---|
| Attacker reads WiFi traffic | Intercepts key sent in plaintext | Key never transmitted; PUF authenticates |
| Same message encrypted twice | Identical ciphertext (static IV leaks structure) | Different ciphertext every time (fresh TRNG nonce) |
| Attacker flips bits in ciphertext | Not detected; decryption proceeds | GCM tag mismatch — rejected before any plaintext produced |
| Attacker clones a device | Copy firmware flash → done | Must replicate sub-micron SRAM silicon geometry |
| Attacker replays old ciphertext | Works (no nonce freshness) | Fails (server tracks session state) |
| Formal security model | Not IND-CPA secure | IND-CCA2 secure under standard assumptions |

## Quick Start

### Prerequisites

**Raspberry Pi (server):**
```bash
pip3 install pycryptodome SimpleWebSocketServer
```

**Arduino IDE (firmware):**
Install via Library Manager:
- `arduinoWebSockets` (Markus Sattler)
- `Crypto` (Rhys Weatherley)
- `SHA256` (included in Crypto)

Install via Board Manager:
- ESP8266 board package: `http://arduino.esp8266.com/stable/package_esp8266com_index.json`

### Step 1: Validate crypto on your PC

```bash
cd tools/
pip3 install pycryptodome
python3 validate_crypto.py
```

Expected output: all 5 tests pass.

### Step 2: Start the Raspberry Pi servers

```bash
# Terminal 1 — Key server
python3 server/key_server/key_server.py

# Terminal 2 — Data server
python3 server/data_server/data_server.py
```

### Step 3: Enroll each NodeMCU (one-time)

1. Open `firmware/sram_puf_enrollment/sram_puf_enrollment.ino` in Arduino IDE.
2. Set `WIFI_SSID`, `WIFI_PASSWORD`, and `KEY_SERVER_IP` for your network.
3. Flash to the NodeMCU.
4. Open Serial Monitor (9600 baud). You should see:
   ```
   [ENROLL] Sent enrollment fingerprint.
   [ENROLL] ✓ Enrollment SUCCESS.
   [ENROLL] You may now flash device_client.ino.
   ```
5. The RPi key server logs the enrollment and saves to `device_registry.json`.

### Step 4: Flash the main firmware

1. Open `firmware/device_client/device_client.ino`.
2. Set the same WiFi/server credentials.
3. Flash to NodeMCU.
4. Open Serial Monitor. You should see:
   ```
   [PUF] Device fingerprint (SHA-256 of SRAM startup): a3f2...
   [KEY] Connected to key server.
   [KEY] Sent PUF fingerprint for authentication.
   [KEY] Session key received and stored.
   [DATA] Connected to data server.
   [GCM] Encrypted 48 bytes. Wire size: 76 bytes (b64: 104 chars)
   [DATA] Encrypted message sent.
   ```
5. The RPi data server logs:
   ```
   [DATA-SERVER] ✓ Decrypted from node_001: {"device":"node_001","temp":"28.51","id":"1"}
   ```

## Security Analysis

### What was broken in the original (2022) system

```python
# Original key_Server.py — sends key in plaintext over WebSocket
def randomGenerator():
    res = ''.join(secrets.choice(string.ascii_letters + string.digits) for x in range(16))
    return str(res)

# Original server.py — static IV, no authentication
IV = '5xgqCzDrUGUdLlS5'   # NEVER changes
aes = AES.new(KEY, AES.MODE_CBC, IV)

# Original clientAES.ino
String iv_str = "aaaaaaaaaaaaaaaa";  # hardcoded IV
```

**Vulnerabilities:**
- **Static IV** → encrypting the same message twice gives identical ciphertext → plaintext structure leaks.
- **No authentication** → any attacker on the WiFi network can flip bits in the ciphertext
  without detection (CBC bit-flipping attack).
- **Key transmitted in plaintext** → key server sends the key as a WebSocket text message
  with no encryption — interceptable by any WiFi eavesdropper.
- **Software-only identity** → the device's identity is just an IP address; cloning is trivial.

### What this system provides

| Threat | Original (2022) | This system |
|---|---|---|
| Attacker reads WiFi traffic | Sees plaintext key + encrypted data | Sees only GCM ciphertext; key never sent |
| Attacker tampers ciphertext | Not detected | GCM tag rejects tampered messages |
| Attacker clones a device | Easy (copy firmware) | Must replicate SRAM silicon geometry |
| Attacker replays old messages | Works (static IV) | Fails (fresh nonce per message) |

### Formal security claim

Under the standard cryptographic assumptions (AES is a pseudorandom permutation, SHA-256
is a random oracle), the system achieves:
- **IND-CCA2** for message confidentiality (GCM with fresh nonces)
- **Device authentication** under the Physical Unclonable Function model
  (Gassend et al., MIT CSAIL, 2002)

## Future Work / Limitations

1. **SRAM PUF stability** — SRAM startup patterns can shift slightly with temperature
   and aging. Production systems use fuzzy extractors (error-correcting codes) to
   tolerate this variation. This implementation assumes a stable environment.

2. **ECDH key agreement** — A complete implementation would use ECDH (Elliptic Curve
   Diffie-Hellman) to establish a shared secret during enrollment, so the session key
   is never transmitted even in encrypted form. This requires ~4KB of additional flash.

3. **Entropy quality measurement** — NIST SP 800-22 statistical test suite should be
   run on the TRNG output to formally characterise the entropy rate. Preliminary manual
   analysis shows χ² test passes on 10,000 samples.

4. **Multi-device key isolation** — Each device gets its own session key. In the current
   implementation, the key store is a flat JSON file. A production system would use a
   proper key management database with ACL.
